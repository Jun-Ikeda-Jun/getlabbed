## 2. プロが求めるもの（プロダクトの核心）

プロは理論を知っている。プロが欲しいのは：

### A. 自分では気づけない癖の統計分析
- 「崖上がりでジャンプ42%、回避35%。Game2以降偏りが増す」
- 「不利状況で暴れ率が高すぎる。この%帯では回避が期待値上」
- 数十試合のデータを蓄積して傾向の変化を追跡

### B. 対戦相手スカウティング
- 大会前に相手の公開試合を分析
- 「Player Xは復帰で上から73%。崖受けではシールド多め」
- 1ページのスカウティングレポートPDFを大会会場でスマホ確認

### C. 適応検出（Anti-Adaptation）
- Bo5セットを通しで分析
- 「Game3で相手がお前の癖に気づいて狩り始めた」
- 「推奨mid-set調整：掴み重視からトマホークに切り替え」

### D. 全89キャラのマッチアップ知識
- 各マッチアップで「何を押し付け、何を突くか」
- フレームデータに裏付けられた具体的な間合い・%帯のアドバイス
- 7,832通りのマッチアップを網羅した知識ベース

## 4. MVPの定義（Phase 1）

### ゴール：「動画を送ったら、プロレベルの試合レビューが返ってくる」を証明する

Phase 1ではCV（Layer 2）を最小限に抑え、マルチモーダルLLM + 知識ベースで価値を出す。

### MVP機能

1. **YouTube URL入力** → 動画取得
2. **キーフレーム抽出**（1秒1フレーム + ストック変動検知で追加抽出）
3. **HUD OCR**（%、ストック → OpenCV + Tesseract。確立された技術）
4. **マルチモーダルLLM分析**（Claude/GPT-4oにフレーム列 + %推移 + 知識ベースを渡す）
5. **試合レビュー出力**：
   - 試合サマリー（勝敗、ストック推移、キーモーメント）
   - 場面ごとの評価（良い択 / 悪い択 / 改善案）
   - マッチアップ別アドバイス
   - 練習メニュー提案

### MVPで「やらない」こと

- リアルタイム分析
- フレーム単位の技認識（CV Layer 2の本格実装）
- 複数試合の傾向追跡（1試合ごとの分析のみ）
- 対戦相手スカウティング
- ユーザー認証・課金

### MVP技術スタック（案）

| レイヤー | 技術 |
|----------|------|
| フロントエンド | Next.js + Tailwind CSS |
| バックエンド | Python (FastAPI) |
| 動画処理 | ffmpeg（キーフレーム抽出）+ OpenCV + Tesseract（HUD OCR）|
| AI分析 | Claude API（マルチモーダル）+ RAG |
| 知識ベース | Ultimate Frame Data（JSON）+ start.gg API統計 |
| ホスティング | Vercel（フロント）+ Railway or Fly.io（バックエンド）|
| 動画ストレージ | Cloudflare R2（S3互換、安い）|

## 7. 法的リスクと対策

### 任天堂リスク（最重要）

任天堂はIP保護で業界最強。以下を厳守：

| ✅ やること | ❌ やらないこと |
|------------|--------------|
| ユーザーがアップロードした動画のみ分析 | ゲーム改造を前提とした機能 |
| サービス名に「Smash」「スマブラ」を入れない | 任天堂の公式アート使用 |
| 「格ゲー動画分析サービス」として位置づけ | YouTube VODの無断ダウンロード |
| キャラ名はファクトとして使用 | リプレイファイルの解析（改造Switch必要） |
| フェアユース範囲内のユーザー生成コンテンツ分析 | ゲーム内データの直接抽出（DRM回避） |

### データソースの合法性

| データ | 方式 | リスク |
|--------|------|--------|
| フレームデータ（UFD） | コミュニティ公開データ参照 | 低（長年運営されている） |
| トーナメント結果 | start.gg公式API | 低（公式API） |
| YouTube字幕 | youtube-transcript-api | 中（YouTube ToS確認要） |
| ゲーム内パラメータ | ultimate-research解析データ | 高（商用利用は要注意） |

## 8. 池田さんが見落としている可能性のある観点

### a. オンライン vs オフラインで最適行動が変わる
- オンライン（入力遅延あり）では反応択が弱くなり、読み択が強くなる
- 分析時に「これはオンライン？オフライン？」を考慮しないと的外れなアドバイスになる

### b. ステージ選択も含めた分析
- 終点と戦場ではキャラの強さが変わる（台ありvs台なし）
- ステージBANの戦略もコーチングの範囲

### c. バージョン（パッチ）対応
- スマブラSPは最終パッチ(13.0.1)で止まっているので問題少ないが
- 将来SF6等に展開する場合、バランス調整パッチへの追従が必須

### d. DI（ベクトル変更）の分析
- コンボを食らっている側のDIが適切かどうかも分析対象
- 「ここでDI外なら生存できていた」は非常に価値あるアドバイス

### e. 「なぜ」を中学生でも分かる言葉で説明する（プロダクトの核心）

専門用語で正確に分析しつつ、説明は誰でも分かる言葉で出す。
「空前を振れ」ではなく、**なぜそれが正解なのか**を体感レベルで伝える。

```
❌ ダメな説明（専門用語の羅列）：
「OoSから発生6Fの上スマで確定反撃。空N着地硬直-8F。」

✅ 良い説明（中学生でも分かる）：
「相手が空中から攻撃してきたね。ガードできたのはナイス！
 でもその後、何もしないでガード解除しちゃった。もったいない！

 実はこの技、ガードした後に相手は少しフリーズするんだ。
 その間に上スマッシュを打てば当たる。撃墜できる%だったから、
 ここで1ストック取れてたかも。

 【練習】トレモでCPUにこの技を振らせて、
 ガード→上スマを10回やってみよう。体が覚えたら試合で自然に出るよ。」
```

ポイント：
- 「何が起きたか」→「なぜそれがもったいないか」→「どうすればよかったか」→「練習方法」の4ステップ
- フレーム数値は裏で使うが、ユーザーには「少しフリーズする」「間に合う」で伝える
- プロには詳細モード（フレームデータ付き）を用意。デフォルトはやさしい説明
- DecodeChess（チェス）の「WHY」説明が最も近い参考モデル

### f. メンタル・対戦心理の領域
- 「同じ択を3回連続で通された後に暴れが増えている」= ティルト検出
- 心理的な崩れパターンの可視化は人間コーチですら見落とす

### g. インフラコスト
- 動画処理は重い。GPU推論コストは1分析あたり約$0.05-0.10
- 月1万分析で$500-1,000。初期はAPI（Claude/GPT-4o）で十分
- スケール後にカスタムモデルで内製化してコスト削減

### h. 英語圏 vs 日本語圏
- 最大市場は英語圏（北米のスマブラ競技人口が最大）
- 日本語UIだけだと市場の80%を逃す
- MVP段階から英語ファーストを推奨
